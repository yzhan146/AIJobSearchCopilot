# PRD: 单 JD 简历优化工作台

## 背景

当前 MVP 已经能跑通简历上传、JD 输入、模型分析和 PDF 生成，但原产品主路径是“提交多个 JD 并排序”。实际求职场景里，更高频的动作是：用户已经看中了一个岗位，需要马上把简历改到更适合这个岗位，并且知道每处修改的原因。

因此下一个 milestone 不再围绕 batch ranking，而是围绕单个目标岗位的简历优化。

## 目标用户任务

用户上传自己的 PDF 简历，提交一个 JD 链接或 JD 描述，然后获得：

1. 面试成功率估计。
2. 简历应该改什么，以及为什么这样改。
3. 一份基于原始事实重新编排和重新叙事的 PDF 简历草稿。
4. 当前与该岗位的差距。
5. 高频面试题和可复用回答思路。
6. 修改后的简历必须保持原始简历语言；中文简历输出中文，英文简历输出英文，其他语言同理。JD 语言只用于理解岗位要求，不能决定简历输出语言。

## 非目标

1. 不再把多个 JD 排序作为主流程。
2. 不再用固定城市、薪资、行业偏好作为默认判断标准。
3. 不允许为了迎合 JD 编造雇主、职位、日期、学历、项目、指标、工具或成就。
4. 不把面试题放在结果第一优先级；它是简历修改之后的辅助输出。
5. 不把中文简历翻译成英文，也不把英文简历翻译成中文，除非用户明确要求翻译版本。

## 核心体验

页面第一屏就是工作台：

1. 左侧提交简历和目标 JD。
2. 右侧展示分析结果。
3. 结果优先级为：成功率、修改记录、PDF 草稿、匹配证据、差距、面试准备。

用户最需要先回答的问题不是“哪个岗位排第一”，而是“这份简历为了这个岗位到底改了哪里，以及这些改动是否可信”。

## 输出结构

服务端主接口为 `POST /api/analyze-target-jd`。

请求包含：

1. `profile`: 简历文件、可选补充文本、网站、GitHub。
2. `job`: 一个目标 JD 的标题、公司、地点、链接、正文。

返回包含：

1. `successProbability`: 0-100 的面试成功率估计。
2. `level`: Strong / Medium / Low。
3. `changeLog`: 修改了什么、原来如何表达、现在如何表达、为什么改、证据来源。
4. `rewrittenResumePdfUrl`: PDF 草稿链接。
5. `resumeRewritePlan`: 简历重排计划。
6. `resumeImprovements`: 可人工检查的改写建议。
7. `gaps`: 当前不足。
8. `interviewQuestions`: 高频面试题和回答。
9. `contentIntegrityNotes`: 不造假检查。
10. `resumeLanguage`: 从原始简历正文检测出的语言，用于约束 PDF 草稿语言。

## RAG Policy

RAG 知识库不应该存放会过期的个人偏好，例如“只看北京”“只看 AI 岗”“固定薪资区间”。这类偏好如果存在，应该来自用户当前请求或可编辑配置，而不是固化进 profile evidence。

本阶段 RAG 分成两层：

1. 用户简历证据 RAG：存放用户真实经历、项目、技能、交付记录、协作背景。
2. 互联网岗位知识 RAG：存放互联网常见岗位族的能力要求、简历强调点、gap 判断和面试关注点。

用户简历证据 RAG 应该存放三类内容：

1. 候选人真实证据：项目、技能、经历、交付记录、协作背景。
2. 改写约束：只能重排、压缩、澄清和重新叙事已有事实，缺失能力必须输出为 gap。
3. 证据选择原则：根据目标 JD 选择最相关证据，而不是根据旧偏好过滤岗位。

互联网岗位知识 RAG 第一版覆盖：

1. 传统产品经理、AI 产品经理、数据产品/BI 产品经理。
2. 后端、前端、全栈、大数据、算法/机器学习。
3. 架构师、解决方案架构师、SRE/DevOps、测试开发、安全/隐私/合规。
4. 项目/交付管理、增长/用户运营/业务运营。

岗位知识只能作为“岗位期待和建议框架”，不能当作候选人事实。所有写进改写后简历的 claim 仍然必须来自用户上传的简历或补充材料。

## 成功标准

1. 用户只提交一个 JD 也能完成完整流程。
2. 页面不再出现多 JD 队列、岗位排名 dashboard、shortlist 等主流程 UI。
3. 每份 PDF 草稿都有可查看的 change log。
4. RAG eval 不再依赖地点和薪资偏好命中。
5. 模型提示词明确要求不编造事实。
